## Context

本项目是一个全新的 AI 工作台，基于 Vue 3 + Vite 构建，部署在 GitHub Pages 上。作为个人工具，需要支持多种 AI 模型接入、Agent 角色管理、文件生成、提示词管理等功能。所有数据存储在客户端（localStorage + IndexedDB），通过 Node.js 本地代理服务器与 AI API 通信，解决跨域和协议适配问题。

**约束条件：**
- 纯前端架构，无自有后端服务器（开发环境使用本地代理）
- 部署在 GitHub Pages（静态站点）
- API Key 由用户自行配置，存在客户端
- 需考虑未来小程序端的兼容性
- 面向中国用户，UI 风格现代化、清爽

## Goals / Non-Goals

**Goals:**
- 构建完整的 AI 对话工作台，支持文字/图片输入、流式输出、文件生成
- 实现多模型、多 Agent 的灵活配置与切换
- 数据完全本地化，支持一键备份/恢复
- 主题系统可扩展，支持日/夜模式
- 通过本地代理服务器解决跨域，支持 Token 和 Coding Plan 两种模式
- 响应式设计，Web 端自适应，手机端体验良好
- 支持思考过程与结论分离显示、联网搜索、图片生成

**Non-Goals:**
- 不做小程序端的具体实现（仅预留适配接口）
- 不做用户系统/登录注册（个人工具）
- 不做服务端数据存储
- 不做多人协作功能
- 不做付费/商业化功能

## Decisions

### 1. 框架选择：Vue 3 + Vite + TypeScript

**选择**: 纯 Vue 3 + Vite 而非 UniApp

**理由**: 
- 项目聚焦 H5 端，UniApp 的跨端能力暂时用不到
- 纯 Vue 3 更轻量、生态更好、开发体验更优
- 避免 UniApp 自定义元素导致的样式穿透问题
- 存储层和 API 调用层已做平台抽象，未来迁移到小程序成本可控

**替代方案**: UniApp — 可跨端但组件渲染有兼容性问题，开发效率低。

### 2. 文件生成方案：结构化 JSON 中间格式

**选择**: AI 输出结构化 JSON → 前端库转换为各格式文件

**理由**: 
- 方案 A（AI 直出解析）效果不可控，复杂格式容易出错
- 方案 B（AI 返回文件）主流 API 不支持
- 方案 C 可控、可调试、可扩展，AI 只负责理解 + 结构化，渲染由代码控制

**技术栈**: SheetJS (Excel)、docx.js (Word)、PptxGenJS (PPT)、jsPDF (PDF)

### 3. 代理层：Node.js 本地代理服务器

**选择**: Node.js + Express 作为本地代理服务器

**理由**:
- 开发环境本地运行，调试方便
- 完美解决跨域问题
- 可做协议适配（Token 模式 vs Coding Plan 模式）
- 支持流式内容提取（思考过程 vs 正文分离）
- 可扩展联网搜索（`/search`）和图片生成（`/image-gen`）端点

**替代方案**: 
- Cloudflare Worker — 需部署，调试不便，冷启动延迟
- 浏览器直连 — 跨域问题无法解决

**生产部署**: 未来可迁移到 CF Worker 或其他 Serverless 平台

### 4. 存储架构：分层存储 + 适配层

**选择**: localStorage (轻量配置) + IndexedDB (大容量数据) + 统一适配层

**理由**:
- localStorage 有 5-10MB 限制，不适合存对话历史
- IndexedDB 容量大（几百 MB），适合存对话历史、图片等
- 适配层屏蔽平台差异，未来小程序端用 uni.setStorage 替换

### 5. 流式输出：fetch + ReadableStream

**选择**: H5 端使用 fetch API 的 ReadableStream

**理由**:
- AI API 通常是 POST 请求，EventSource (SSE) 只支持 GET
- fetch Stream 支持 POST + 流式读取
- 浏览器原生支持，零依赖

**小程序端**: 预留接口，未来用 wx.request + enableChunked 实现

### 6. 思考过程与结论分离

**选择**: 代理服务器识别思考过程事件，前端分别渲染

**理由**:
- 火山方舟 Responses API 的思考过程通过 `response.reasoning_summary_text.delta` 事件返回
- 代理服务器提取思考内容和正文内容，分别返回给前端
- 前端用不同样式渲染：思考过程小字（11px）、可折叠；结论正常字号（15px）

### 7. 联网搜索：查询重写 + Bing 搜索

**选择**: AI 查询重写 → Bing 搜索 → 结果注入上下文

**理由**:
- 用户原始消息不适合直接搜索（如"天津今天天气怎么样"）
- 先用 AI 提取关键词（"天津天气预报"），提高搜索精度
- Bing 搜索无需 API Key，解析 HTML 返回结构化结果
- 搜索结果注入用户消息，AI 基于搜索结果回答

### 8. 图片生成：专用模型 + 开关模式

**选择**: 画图开关按钮 + 专用图片生成模型（`doubao-pro-3_0`）

**理由**:
- 文本模型（如 `deepseek-v4-pro-260425`）不支持图片生成
- 使用专用图片生成模型，通过 `/image-gen` 端点调用
- 开关模式让用户明确当前是画图还是对话

### 9. 主题系统：CSS 变量注册表

**选择**: 基于 CSS 自定义属性（CSS Variables）的主题引擎

**理由**:
- 运行时切换，无需重新编译
- 注册表模式，新增主题只需注册一组变量值
- Vue 3 的 SCSS 原生支持 CSS 变量
- 性能好，切换只是替换变量值

### 10. 模型调度：三级优先级

**选择**: 全局默认 → Agent 覆盖 → 会话临时

**理由**: 满足不同粒度的需求——全局统一配置、Agent 个性化、临时实验。"一键统一"功能将 Agent 级覆盖清空，回退到全局默认。

### 11. 数据导出/导入：JSON 文件 + 并集合并

**选择**: 导出为 JSON 文件，导入时与本地数据取并集

**理由**:
- JSON 通用、可读、易调试
- 并集策略避免覆盖本地独有数据
- 冲突以文件为准（用户主动导入的意图明确）
- 对话历史可选包含（避免文件过大）

### 12. Key 加密：AES-GCM via Web Crypto API

**选择**: 浏览器原生 Web Crypto API，AES-GCM 算法

**理由**:
- 浏览器原生支持，无需第三方库
- AES-GCM 提供加密 + 完整性校验
- 密码不持久化，关闭页面后需重新输入
- 可选功能，用户也可选择明文存储

### 13. 提示词库：标签分类 + 全文搜索

**选择**: 标签（多标签）+ 全文搜索 + 变量模板

**理由**:
- 标签比文件夹更灵活，一个提示词可属于多个分类
- 搜索覆盖标题、内容、标签
- 变量模板（`{{变量名}}`）让提示词可复用
- 使用次数统计支持排序

## Risks / Trade-offs

- **[IndexedDB 兼容性]** → 使用成熟的封装库（如 idb 或 Dexie），并在适配层做降级处理
- **[本地代理服务器部署]** → 开发环境本地运行，生产环境需迁移到 Serverless 平台（如 CF Worker）
- **[小程序 storage 上限 10MB]** → 当前聚焦 H5，小程序端未来可通过裁剪对话历史、压缩数据应对
- **[API Key 前端存储安全性]** → 个人工具可接受，加密功能降低风险；开源代码不含实际 Key
- **[文件生成库体积]** → SheetJS、docx.js 等库较大，需按需加载（动态 import）
- **[Bing 搜索 HTML 解析稳定性]** → HTML 结构可能变化，需定期维护解析逻辑
- **[图片生成模型额度]** → 用户需确认 Coding Plan 包含图片生成模型额度
- **[GitHub Pages 首次加载速度]** → 通过代码分割、懒加载、CDN 缓存优化
