## Context

当前项目是一个纯前端 AI 工作台（Vue 3 + Pinia + IndexedDB），Agent 系统已实现单智能体的 CRUD 和对话集成。所有 AI 调用通过本地代理（Express）或 Cloudflare Worker 转发，本质是 HTTP POST 请求。

现有架构特点：
- 纯前端，无后端服务
- 代理层只做请求转发，不维护状态
- 对话通过 `sendChat` / `sendChatStream` 发起，支持流式和非流式
- Agent 数据存储在 IndexedDB，通过 Pinia store 管理

## Goals / Non-Goals

**Goals:**
- 实现顺序流水线（Pipeline）：用户定义 Agent 执行顺序，系统串行执行并传递上下文
- 实现主从模式（Commander）：指挥官 Agent 自动拆解任务，分配给执行者，最后汇总
- 协作过程可视化：展示每个步骤的执行状态、中间结果
- 完全在前端实现，不依赖后端新增接口
- 现有单 Agent 对话功能零影响

**Non-Goals:**
- 不实现 Agent 之间的实时通信或并行执行
- 不实现 Agent 市场或团队模板共享
- 不修改代理层（proxy）的逻辑
- 不实现跨设备的协作团队同步

## Decisions

### 决策 1：统一数据模型 — Crew 同时承载流水线和主从模式

**选择**：用一个 `Crew` 类型统一表示，通过 `mode` 字段区分 `pipeline`（流水线）和 `commander`（主从）。

**理由**：
- 两者本质都是"一组 Agent 协作完成任务"，只是编排方式不同
- 统一模型减少 UI 复杂度，用户在一个页面管理所有协作团队
- 避免创建两套几乎平行的数据结构和页面

**替代方案**：分别创建 Pipeline 和 Commander 两个独立类型 → 导致 UI 和 Store 重复代码多，用户体验割裂。

### 决策 2：执行引擎在前端实现

**选择**：协作执行逻辑完全在前端（Store 层）实现，通过多次调用现有 `sendChat` / `sendChatStream` 完成。

**理由**：
- 每次子任务执行本质就是一次普通 API 调用，只是 systemPrompt 和上下文不同
- 代理层无需感知协作逻辑，保持简单
- 前端可以精确控制执行顺序、进度展示、中断恢复

**替代方案**：在代理层实现协作编排 → 增加后端复杂度，且前端无法精细控制 UI 展示。

### 决策 3：串行执行，不支持并行

**选择**：所有子任务严格按顺序执行。

**理由**：
- 前端并发请求 + 多流式响应同时处理的复杂度极高
- 串行执行更可控，用户能清晰看到每一步的进展
- 大多数协作场景下，后续步骤依赖前序结果，并行收益有限

### 决策 4：上下文传递采用"摘要传递"策略

**选择**：每个步骤执行完后，将输出结果作为文本上下文传递给下一个步骤的 systemPrompt 或用户消息中。对于主从模式，指挥官拆解时声明每个子任务需要的前置结果。

**理由**：
- 全量传递会导致上下文快速膨胀，Token 消耗不可控
- 按需传递需要指挥官输出结构化指令，实现复杂
- 摘要传递在 Token 消耗和信息完整性之间取得平衡

### 决策 5：协作结果存储为单条对话消息

**选择**：协作完成后，将最终汇总结果作为一条 assistant 消息存储，中间步骤结果作为消息的 `crewResult` 附件字段（类型为 `CrewResult`，含 `steps` 数组和 `finalOutput`）。

**理由**：
- 不污染现有的消息列表结构
- 用户可以展开查看中间步骤，但不强制显示
- 与现有存储层兼容，无需修改 IndexedDB schema

### 决策 6：分阶段实施

**选择**：先实现 Pipeline 模式（阶段一），验证协作流程可行后，再叠加 Commander 模式（阶段二）。

**理由**：
- Pipeline 实现简单（固定顺序串行调用），可快速验证核心链路
- Commander 依赖 Pipeline 的执行基础设施（步骤管理、进度展示、上下文传递）
- 降低一次性交付风险

### 决策 7：协作模式与辅助功能互斥

**选择**：选择 Crew 执行协作时，自动禁用联网搜索和图片生成开关。图片上传保留（作为第一个 Agent 的输入）。

**理由**：
- 联网搜索的结果注入到流水线哪一步不明确，语义冲突
- 图片生成与协作模式的目标不同，同时启用会造成混淆
- 图片上传作为输入是合理的（如"分析这张图片"交给流水线处理）

### 决策 8：加密 API Key 预检查

**选择**：协作执行前，预检查所有步骤涉及的模型 API Key 是否可用（未加密或已解密）。不可用时提示用户先解锁。

**理由**：
- 协作执行是连续过程，中途弹出解密对话框会打断流程
- 提前检查避免执行到一半才失败

### 决策 9：指挥官拆解输出格式

**选择**：通过指挥官的 systemPrompt 约束其输出固定 JSON 格式，包含子任务列表（每个子任务含 description、agentId）。前端解析该 JSON 生成执行计划。

**理由**：
- JSON 格式便于前端解析和渲染
- 通过 systemPrompt 约束而非代码解析，灵活适配不同模型
- 解析失败时回退到纯文本展示，不阻塞执行

### 决策 10：协作执行期间锁定输入

**选择**：协作执行期间禁用发送按钮，显示"协作执行中..."提示。用户仍可点击停止按钮中断执行。

**理由**：
- 避免并发执行多个协作导致状态混乱
- 用户仍可中断当前执行

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| Token 消耗成倍增加（3-5 倍） | 展示预估消耗提示；限制子任务数量上限（建议 3-5 个）；上下文传递采用摘要策略 |
| 串行执行总耗时长 | 每个步骤展示进度和预计状态；支持中途取消；步骤结果实时展示减少等待焦虑 |
| 指挥官拆解质量不稳定 | 提供拆解结果确认步骤（用户可调整后再执行）；指挥官提示词精心设计 |
| 中间步骤结果过长导致上下文溢出 | 对传递的上下文做截断处理（限制最大字符数） |
| 协作失败时用户体验差 | 单步失败时展示错误信息，支持从失败步骤重试；已完成的步骤结果保留 |
