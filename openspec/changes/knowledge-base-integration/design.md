## Context

当前项目为纯前端架构（Vue 3 + Pinia + IndexedDB），无后端服务。Agent 系统已实现单智能体和协作模式，但缺乏外部知识接入能力。药学场景需要处理大体积文档（药典、法规），对检索精度和溯源要求极高。

**约束条件**：
- 纯前端实现，所有数据存储和检索在浏览器端完成
- IndexedDB 存储上限约 1-2GB，足够存储药典等文档（纯文本约 50-100MB）
- 不能依赖外部向量数据库或 Embedding API（成本和隐私考虑）
- 需要支持大文档（10 万 + 字符）的高效检索

## Goals / Non-Goals

**Goals:**
- 支持上传 .txt/.md 文档，自动解析结构并分块存储
- 实现基于目录索引 + 全文检索的混合检索机制
- Agent 可绑定多个知识库，对话时自动检索并注入相关上下文
- 检索结果标注来源（文档名、章节、位置），支持查看原文和下载
- 优化 Token 使用：摘要预生成 + 分级注入 + 对话级缓存

**Non-Goals:**
- 不支持 PDF/Office 文档解析（MVP 阶段，后续扩展）
- 不使用向量检索/Embedding（纯前端全文检索足够）
- 不支持多用户共享知识库（本地存储，单用户）
- 不支持实时协作编辑知识库

## Decisions

### 决策 1：文档分块策略 — 结构化分块（按章节/条目）

**选择**：按文档自然结构（标题层级）分块，而非固定字数切分

**理由**：
- 药典/法规本身高度结构化（章节→条目→段落）
- 按结构分块保持语义完整性，避免截断
- 支持目录索引快速定位，无需遍历所有块

**替代方案**：
- 固定字数切分（500 字/块）：简单但破坏语义结构
- 滑动窗口：改善覆盖率但仍可能截断条目

### 决策 2：检索引擎 — flexsearch（全文检索）

**选择**：使用 flexsearch 库实现客户端全文检索

**理由**：
- 纯前端实现，无需后端
- 支持中文分词（需配置）
- 性能优秀（百万级文档毫秒级检索）
- 体积小（~10KB gzipped）

**替代方案**：
- lunr.js：类似功能，但 flexsearch 性能更好
- 自建倒排索引：开发成本高，无必要

### 决策 3：存储架构 — 三级存储（知识库→文档→块）

**选择**：IndexedDB 中建立三级存储结构

```
KnowledgeBase (id, name, description, createdAt)
── Document (id, knowledgeBaseId, title, source, structure, createdAt)
│   └── Chunk (id, documentId, title, content, summary, keywords, position, createdAt)
└── Index (documentId, flexsearch 索引数据)
```

**理由**：
- 层级清晰，便于管理
- 块级别存储支持细粒度检索
- 索引与数据分离，便于重建

### 决策 4：摘要预生成策略 — 上传时生成

**选择**：文档上传时调用 LLM 为每个块生成摘要（一次性成本）

**理由**：
- 检索时注入摘要而非原文，节省 60-80% Token
- 摘要只生成一次，后续检索零成本
- 上传时用户有心理预期（"正在处理文档..."）

**替代方案**：
- 首次检索时生成：延迟高，用户体验差
- 不生成摘要：每次注入原文，Token 消耗大

### 决策 5：检索结果注入 — 分级注入 + 阈值控制

**选择**：
1. 检索返回 Top 5 候选块
2. 计算相关度分数（关键词匹配度 + 位置权重）
3. 只注入分数 > 0.6 的块
4. 最多注入 3 块（硬上限）
5. 优先注入摘要，用户追问时注入原文

**理由**：
- 避免无关内容浪费 Token
- 控制注入内容在 500-1500 字（400-2000 tokens）
- 分级注入按需加载，进一步优化

### 决策 6：Agent 绑定知识库 — 多对多关联

**选择**：Agent 可绑定多个知识库，对话时自动加载所有关联知识库

**理由**：
- 药学助手可能需要同时查药典 + 法规
- 用户选择 Agent 后无需手动选知识库
- 支持对话中动态添加/移除

### 决策 7：来源溯源 — 消息元数据 + 原文面板

**选择**：
- AI 消息新增 `knowledgeSources` 字段，记录引用的块 ID
- 回答底部显示来源卡片（文档名、章节、位置）
- 点击来源卡片弹出原文面板（完整条目内容）
- 提供"下载条目"按钮（导出为 Word）

**理由**：
- 增强可信度，用户可验证信息
- 原文面板避免主对话区过长
- 下载功能方便离线使用

## Risks / Trade-offs

### [风险] 中文分词效果不佳
**缓解**：flexsearch 支持自定义分词器，可集成 jieba 分词或按标点/空格简单分词。药学专业术语可通过关键词标签补充。

### [风险] 大文档上传处理时间长
**缓解**：分块和摘要生成在 Web Worker 中执行，避免阻塞 UI。显示进度条（"正在处理文档... 30%"）。

### [风险] IndexedDB 存储空间不足
**缓解**：单知识库上限 500MB，超限时提示用户。提供知识库清理功能（删除旧文档）。

### [风险] 检索结果不相关
**缓解**：阈值控制（< 0.6 不注入）+ 用户反馈机制（"结果是否有用？"）优化排序。

### [权衡] 摘要质量依赖 LLM
**接受**：摘要用于快速预览，用户可查看原文验证。摘要生成失败时降级使用原文前 200 字。

### [权衡] 纯全文检索无语义理解
**接受**：药学查询通常是精确匹配（药名、剂量），全文检索足够。后续可扩展向量检索作为补充。
